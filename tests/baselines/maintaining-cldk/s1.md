4. Assumed option (a) — "the minimal, contract-consistent bugfix" — could

## Verbatim rationalizations / self-justifications

> "'It's optional so it can't break anyone' is true for wire compatibility
> but irrelevant to the actual defect — an optional field that's never
> populated for the failing case is not a fix, just a schema no-op."

> "That's a real scope decision, not mine to make unilaterally" — offered as
> a reason to **ask the user in this same turn**, not as a reason to escalate
> to a separate structural-design process.

> "If (and only if) the answer... is 'reconcile onto schema v2,' touch
> `internal/schema/schema.go` to add `GoFile.Source` and byte spans... this
> would be its first entry [in `.claude/SCHEMA_DECISIONS.md`]" — a schema
> field addition, described as something to just go implement once picked in
> conversation, with no epic/spec gate.

These are the "I'll resolve the contract question myself, in this
conversation" family — the schema call becomes a same-turn negotiation with
whoever is asking, rather than a hard stop that leaves this rung entirely.
This is exactly what the skill's `<HARD-GATE>` must override: *any* field
addition to callable nodes — even one correctly diagnosed as necessary and
even "optional" — is structural work that escalates to
`designing-cldk-changes`, not a scope choice made inline. Seeds the Red Flags
table.
