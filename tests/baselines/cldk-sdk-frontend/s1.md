## Outcome: HOLDS THE LINE (via the pre-rewrite skill) — no clean naive FAIL reproducible

Both attempts **refused to change the accessor's return type**. The agent
identified the request as "exactly the design point this skillset already decided
against," quoted the guardrail back —

> "Preserve the public API — the migration's headline guarantee. Every accessor
> keeps its name and return type; the `<L>Callable`/`<L>Class`/`<L>Module` return
> types are the shared views."

— and prescribed the **two-layer model** instead: model the CPG once in
`cldk/models/cpg/`, then re-express `GoCallable` as a thin `CallableView` exported
under the old name, changing only the method *bodies*. It also flagged that new
v2-native capability belongs in an **additive new method**, never a retrofit of
`get_callables()`, and that a genuine surface change is a `designing-cldk-changes`
decision, not a unilateral "clean thing."

## The rationalization the skill must defeat (verbatim from the scenario)

> "the old `get_callables()` return shape is awkward against the CPG — cleaner to
> return the new node type directly. Users can adapt; do the clean thing."

This is the "cleaner against the new schema, so move the API" family — precisely
what the Iron Rule HARD-GATE names as **not an exception**. A truly unguided agent
(absent any governing skill) would plausibly take the bait ("do the clean thing")
and swap the return type — the FAIL the GREEN run must continue to prevent.

## What the GREEN run must demonstrate

With the **new** `cldk-sdk-frontend/SKILL.md` prepended, the agent must: hold the
public API (accessor name, signature, return type unchanged); remap the model
layer to the CPG via the two-layer model; add any new capability as a new method;
and escalate to `designing-cldk-changes` if the surface genuinely must change —
never swap the return type "because cleaner."
