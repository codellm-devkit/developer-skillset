an SDK against an analyzer that does not yet emit conformant output.
Prompt (verbatim): "Wire Lua into python-sdk. The analyzer is half-done but the
models can be stubbed from the schema doc."

## Contamination note (honest record — read first)

As with S1, a clean *unguided* RED was **not achievable in this environment**: the
`cldk-forge` plugin (the pre-rewrite `cldk-sdk-frontend`) is installed
session-wide and the workspace holds the skill source. Two dispatch attempts:

1. **Brief cwd** (`/Users/rkrsn/workspace/codellm-devkit`). The agent discovered
   and quoted the skill's "Precondition is real" guardrail.
2. **Isolated cwd** (an empty scratchpad `red-s2/workspace`). The agent still
   found the real workspace and **explicitly "Loaded the `cldk-sdk-frontend`
   skill"**, then applied its precondition guardrail.

The prompt is unmistakably CLDK ("Wire Lua into python-sdk", "the schema doc"), so
the agent located the governing skill regardless of cwd. The pre-rewrite skill
already carries the precondition gate, so the agents refused; the RED documents
the rationalization and confirms the gate is load-bearing, but cannot show a naive
FAIL because a guardrailed skill was unavoidably present.

## Outcome: REFUSES (via the pre-rewrite skill) — no clean naive FAIL reproducible

Both attempts **refused to stub models against an unshipped analyzer**. The agent
verified there is no `codeanalyzer-lua` anywhere (not in the workspace, not on the
`codellm-devkit` GitHub org — `gh repo view codellm-devkit/codeanalyzer-lua`
404s), no sample `analysis.json`, no `SCHEMA_DECISIONS.md`, no version to pin. It
quoted the guardrail —

> "Precondition is real. … If `codeanalyzer-<lang>` isn't emitting conformant
> `analysis.json` yet, stop and run codeanalyzer-backend first — you can't
> validate SDK models against output that doesn't exist."

— and named the specific harm of stubbing: Pydantic silently drops unknown keys,
so any real Lua-native field would load-and-vanish with no fixture to catch it,
and the "definition of done" checks (dangling-free `get_call_graph()`, the backend
contract test, E2E) are structurally unsatisfiable without a real analyzer. It
routed to **codeanalyzer-backend** as the correct next rung.

## The rationalization the skill must defeat (verbatim from the scenario)

> "The analyzer is half-done but the models can be stubbed from the schema doc."

This is the "the schema doc is enough, we can validate later" family — the exact
anti-pattern the entry precondition names. A truly unguided agent (absent any
governing skill) would plausibly scaffold `cldk/models/lua/` from the generic
schema doc and call it wired — the FAIL the GREEN run must continue to prevent.

## What the GREEN run must demonstrate

With the **new** `cldk-sdk-frontend/SKILL.md` prepended, the agent must: refuse to
start; state that the entry precondition (a working, schema-conformant
`codeanalyzer-lua` emitting a real `analysis.json`) is unmet; and route back to
**codeanalyzer-backend** — never stub SDK models against the schema doc.
