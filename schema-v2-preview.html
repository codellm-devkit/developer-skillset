The CLDK analysis schema, v2

Codeanalyzer is an additive analysis paradigm: each analysis level is the same tree grown one layer deeper, plus one edge family over the new layer. One structure — a containment tree with typed edge overlays — is a Code Property Graph. Symbol table, call graph, CFG, PDG, SDG, taint: all projections of it.

One atom: a scale-free node — id · kind · span · children · edges Two relations: containment (single-parent) + typed edges (many) Two projections: analysis.json + Neo4j

The tree, grown level by level

The worked example from the spec — Hasher.Hash calling fnv.New64 in a Go app. Step through the levels: every step only adds (or refines null → id); nothing is ever rewritten. Toggle edge families to isolate an overlay.

Intra-callable edges (cfg·cdg·ddg·summary) hang on the callable; cross-callable edges (call_graph·param_in·param_out) hang on the application — always the lowest common ancestor of their endpoints.

The level ladder

Depth grows only at L1 and L3; L2 is pure refinement + one edge family; L4 adds synthetic vertices + cross edges. L1 ⊆ L2 ⊆ L3 ⊆ L4, superset modulo null-refinement — a CI-checkable gate. Framework enrichment (Joern/WALA) is an orthogonal axis, provenance-merged into existing families, never a level.

Identity: two tiers, split at the callable leaf line

The boundary is exactly where L1 stops. Above it, ids are durable; below it, they are ordinal within one analysis run — cross-edit durability below the callable is a declared non-goal.

Durable ids  ≥ callable

Containment-path URIs with an app segment; survive re-analysis; what SCIP export and cross-repo joins address.

can://<lang>/<app>/<file>/<type>/<callable-signature>
can://go/myapp/src/util.go/Hasher/Hash(string)uint64

Ordinal ids  < callable

Addressed within their callable: a source position for real nodes, an @tag for synthetic ones. Always keep the column — a bare line is not unique.

<callable-id>@<line>:<col> — …/Hash(string)uint64@15:2
<callable-id>@<tag> — …@entry · …@formal_in:0 · …@16:2/actual_in:0

The invariants that make “additive” true

  • Monotonicity. Levels add facts, never contradict or delete. Two sanctioned changes only: new nodes/edge entries, and the callee: null → id refinement. The DDG chain holds because the posture is weak-update/over-approximate: L3's syntactic def-use is a strict subset, L4 adds alias-derived edges — a strong update would remove one and break ⊆.
  • One source blob per module. Text lives once, on the module's source; every node's text is a slice by its span. Spans carry byte offsets alongside line:col so slicing is O(1). Subsumes any per-callable code field.
  • Present or absent — no null (except the sanctioned callee slot). Absence is the “no fact” encoding.
  • snake_case keys everywhere, in every host language, so one set of SDK models parses every analyzer. max_level in the payload declares what was populated — consumers read it, never sniff for keys.
  • Cross-language parity. Node kinds, edge-list names, edge kind/prov values are identical across analyzers. A language adds kinds (Go defer_resume, Rust unsafe, TS interface) in its SCHEMA_DECISIONS.md — it never renames or repurposes a shared name. This is what lets the SDK model the schema once: one Node, one Edge, one Application.

Two projections, one structure

analysis.json

The facade contract — what the SDK models validate against.

Named-map tree (symbol_table → types → callables), body maps keyed by local id, split edge lists, source per module. Levels gate this path only (-a 1…4).

neo4j

Near-identity co-primary projection — a first-class deliverable, not an afterthought.

Every node → a node row; containment rendered as typed HAS_*/DECLARES edges; every overlay edge → a typed relationship. Always full depth — the analyzer runs at maximum implemented level when the target is the graph.
